# Outbound

An `Outbound` enables **egress via SNAT**. It allocates IP addresses from a
[`Network`](../getting-started/concepts.md#foundation-vrf-and-network) and
publishes them as **Coil Egress** IPs and **Calico IPPools**, so pods can reach
external destinations from a stable, well-known source address range.

Which data plane you get depends on a single field, `destinations`:

- **HBN mode** (`destinations` set) — additionally creates VRF `/32`/`/128`
  host exports, egress NAT, and policy-based routing (source-based routing, SBR)
  into the matched VRFs.
- **non-HBN mode** (`destinations` omitted) — creates only the Coil Egress and
  Calico IPPool objects.

## How it works

The controller resolves `networkRef` to a `Network` and allocates addresses from
its pool (a `count` of IPs, or — less commonly — explicit `addresses`). From that
allocation the operator's **Coil platform controller** (`platform-coil`, part of
this operator) automatically creates and reconciles, for each `Outbound`:

- **Calico IPPools** — one host pool (`/32` / `/128`) per allocated address.
- a **Coil `Egress`** (`coil.cybozu.com/v2`) named after the `Outbound`, whose
  gateway pods SNAT egress traffic and source it from those pools.
- an **egress `NetworkPolicy`** for the gateway pods.

!!! note "You do not author the Coil Egress"
    The Coil `Egress`, Calico `IPPool`s and `NetworkPolicy` are **generated by
    this operator** from the `Outbound` — you never write them by hand. Your only
    job is to create the `Outbound` and then point your application pods at the
    generated `Egress` (see [Using the egress from your pods](#using-the-egress-from-your-pods)).

When `destinations` is set (**HBN mode**), the matched `Destination` CRDs resolve
to one or more VRFs (`Destination.spec.vrfRef`). The controller then also:

- exports the allocated addresses as **host routes** (`/32` / `/128`) into those
  VRFs,
- programs **egress NAT**, and
- installs **policy-based routing (SBR)** so traffic from the allocated source
  addresses is steered into the matched VRFs.

When `destinations` is omitted (**non-HBN mode**), no VRF plumbing is performed —
only the Coil Egress and Calico IPPool objects are created.

!!! note "Mode is decided by field presence"
    There is no mode switch. `Outbound` has no `interfaceRef`; the mode is
    decided **solely** by whether `destinations` is set. See
    [Deployment modes: HBN vs. non-HBN](../getting-started/concepts.md#deployment-modes-hbn-vs-non-hbn-pure-l2-netplan).

## Prerequisites

- A [`Network`](../getting-started/concepts.md#foundation-vrf-and-network)
  referenced by `networkRef`. This provides the address pool.
- For **HBN mode**: at least one
  [`Destination`](../getting-started/concepts.md#routing-destination) matched by
  `destinations`, bound to a `VRF` via its `vrfRef`.
- **Coil** and **Calico** present in the cluster for egress to actually work.
  The `Outbound` (via this operator's `platform-coil` controller) creates the
  Coil `Egress` and Calico `IPPool`s; the Coil runtime provides the egress
  gateway pods. See [Using the egress from your pods](#using-the-egress-from-your-pods).

See the [Concepts](../getting-started/concepts.md) page for the resource model
and the [Quick Start](../getting-started/quick-start.md) for creating the
foundation resources.

## Minimal example

A dual-stack egress allocation in HBN mode. The controller allocates `count` IPs
from the `Network` and exports them into the VRFs behind `Destination`s labelled
`type: gateway`:

```yaml
apiVersion: network-connector.sylvaproject.org/v1alpha1
kind: Outbound
metadata:
  name: ob-egress
  namespace: default
spec:
  networkRef: "net-vlan501"
  destinations:
    matchLabels:
      type: gateway
  count: 2
```

To run in **non-HBN mode** (Coil Egress + Calico IPPool only), omit the
`destinations` block.

!!! tip "Prefer `count`"
    Allocating from the `Network` with `count` is the recommended, predominant
    model — you let the operator pick free addresses from `networkRef`. Only use
    explicit `addresses` when you must pin specific IPs (see
    [Pinning specific addresses](#pinning-specific-addresses)).

## Field reference

Key `spec` fields:

| Field | Type | Required | Notes |
|---|---|---|---|
| `networkRef` | string | yes | Name of the `Network` to allocate from. **Immutable**. |
| `destinations` | label selector | no | Selects `Destination` CRDs. **Presence toggles HBN (set) vs. non-HBN (omitted)**. |
| `count` | integer (≥1) | conditional | **Recommended.** Number of IPs to allocate from the `Network`. **Mutually exclusive with `addresses`**. |
| `addresses` | object | conditional | Explicit allocations with `ipv4` and/or `ipv6` CIDR lists. Use only when you must pin specific IPs. **Mutually exclusive with `count`**. |
| `replicas` | integer (≥1) | no | Number of Coil egress pod replicas. |

!!! warning "Exactly one of `count` or `addresses`"
    The admission webhook enforces that **exactly one** of `count` or
    `addresses` is set. Setting both, or neither, is rejected. Prefer `count`.

For the full, authoritative field list (including `status`), see the
[CRD Reference](../reference/crd-reference.md#outbound).

## Using the egress from your pods

This operator's `platform-coil` controller creates a Coil `Egress` **named after
the `Outbound`**, in the same namespace, backed by the Calico IPPools it
generates for the allocated addresses. You do **not** create the `Egress`,
IPPools or NetworkPolicy yourself.

To send an application pod's traffic out through that egress, annotate the pod
(this is the only Coil-facing resource you author):

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: my-app
  namespace: default
  annotations:
    # key:  egress.coil.cybozu.com/<egress-namespace>
    # value: the Outbound name (== the generated Egress name)
    egress.coil.cybozu.com/default: ob-egress
spec:
  containers:
    - name: app
      image: busybox:1.37
      command: ["sleep", "infinity"]
```

!!! note "Egress name = Outbound name"
    The generated `Egress` shares the `Outbound`'s name and namespace. The
    number of egress gateway pods follows `spec.replicas` (default `1`), and the
    egress `destinations` are the prefixes from the matched `Destination`s.

## Common tasks

### Allocate with `count` (recommended)

Let the controller pick free addresses from the `Network` — the predominant
model:

```yaml
spec:
  networkRef: "net-vlan501"
  destinations:
    matchLabels:
      type: gateway
  count: 4
```

### Pinning specific addresses

Only when you must pin exact IPs, use `addresses` (mutually exclusive with
`count`):

```yaml
spec:
  networkRef: "net-vlan501"
  destinations:
    matchLabels:
      type: gateway
  addresses:
    ipv4:
      - "10.250.7.0/30"
    ipv6:
      - "fd90:4dbf:396f::/126"
```

### Dual-stack

Use `count` against a dual-stack `Network` to allocate from both families at
once, or (when pinning) provide both `ipv4` and `ipv6` lists under `addresses`.

### Setting `replicas`

Scale the number of Coil egress gateway pod replicas:

```yaml
spec:
  networkRef: "net-vlan501"
  destinations:
    matchLabels:
      type: gateway
  count: 2
  replicas: 3
```

### Single-VRF SBR egress via the destination selector

Target a single VRF by scoping the `destinations` selector:

```yaml
apiVersion: network-connector.sylvaproject.org/v1alpha1
kind: Outbound
metadata:
  name: ob-sbr-single
spec:
  networkRef: "net-vlan501"
  destinations:
    matchLabels:
      type: gateway
  count: 1
```

The set of VRFs an Outbound plumbs into is whatever the matched `Destination`s
resolve to — narrow or widen the selector to control it.

## Verify

List Outbounds (the `ob` short name also works):

```bash
kubectl get outbound
kubectl get ob
```

Inspect the allocated addresses, the VRFs, and the conditions:

```bash
kubectl get outbound ob-egress -o jsonpath='{.status.addresses}' | jq
kubectl get outbound ob-egress -o jsonpath='{.status.vrfs}' | jq
kubectl get outbound ob-egress -o jsonpath='{.status.conditions}' | jq
```

- `status.addresses` — the IPs actually allocated (`ipv4` / `ipv6`).
- `status.vrfs` — the VRFs plumbed into (populated in HBN mode; empty in
  non-HBN mode). Sorted and de-duplicated.
- `status.conditions` — readiness, chiefly `Ready` and `Resolved`.

## Troubleshooting

!!! warning "`Resolved=False` — Network or Destination missing"
    If `networkRef` points to a non-existent `Network`, or the `destinations`
    selector matches no `Destination`, the `Resolved` condition stays `False`
    and no allocation is made. Confirm the `Network` exists and the
    `Destination` labels match your selector.

!!! warning "`networkRef` is immutable"
    `networkRef` cannot be changed after creation — the allocations derived from
    it cannot be rebound in place. To move to a different `Network`, delete and
    recreate the `Outbound`.

!!! warning "`count`/`addresses` exclusivity"
    Setting both `count` and `addresses`, or neither, is rejected by the
    admission webhook. Set exactly one.

!!! tip "Egress not working"
    The `Outbound` and the operator create the Coil `Egress`, Calico `IPPool`s
    and `NetworkPolicy`, but actual SNAT egress requires the **Coil** and
    **Calico** runtimes in the cluster. If pods are not egressing:

    - verify Coil and Calico are installed and healthy,
    - confirm the generated `Egress` exists:
      `kubectl get egress.coil.cybozu.com <outbound-name> -n <namespace>`,
    - check your application `Pod` has the
      `egress.coil.cybozu.com/<namespace>: <outbound-name>` annotation.
